# Maestro의 아키텍처 가치: 조합 가능한 시스템 설계

## 핵심 원칙: 기능 추가가 아닌 플로우 조합

### 실제 사례: 댓글 → 템플릿 자동 생성

"댓글을 분석해서 답변 템플릿을 만드는" 기능은 새로운 비즈니스 로직 없이 구현되었습니다.

**방법**: 기존 "댓글 조회" 플로우와 "템플릿 생성" 플로우를 어댑터로 연결

**결과**: "Search comments and create reaction template" 명령어가 자동으로 작동

→ **핵심 가치**: 기존 컴포넌트를 재조합하여 새로운 기능을 만드는 구조

---

## 1. 플로우 체이닝: 기능의 레고 블록화

### 어댑터 보너스 메커니즘

시스템이 복합 쿼리를 분석할 때, 어댑터로 연결 가능한 플로우 조합에 **자동으로 우선순위를 부여**합니다.

**작동 방식**:
- 사용자: "트렌드 찾고 → 그걸로 초안 작성해줘"
- 시스템: "트렌드 조회" + "초안 생성" 플로우 조합 감지
- 자동 실행: 트렌드 데이터를 초안 생성용 파라미터로 변환

### 현재 구현 범위

**6개 어댑터 패턴**으로 **134개 플로우** 간 복합 기능 지원:
- 트렌드 → 초안 생성
- 댓글 → 초안 생성
- 댓글 → 답변 템플릿 생성
- 타임라인 결과 연속 조회

### LLM 역할의 재정의

**기존 AI 제품**: LLM이 모든 결정과 실행 담당 → 불안정성, 비용 증가

**Maestro 방식**:
- **시스템**: 라우팅, 데이터 변환, 플로우 조합 결정
- **LLM**: 고품질 콘텐츠 생성만 담당
- **Fallback**: LLM 실패 시 규칙 기반 대안 자동 제공

→ **비즈니스 가치**: 예측 가능한 동작 + LLM 비용 최적화

---

## 2. Graph RAG: 관계 기반 지식 관리

### 차별점: 벡터 검색의 한계 극복

**일반 RAG (벡터 검색만)**:
- "AI 트렌드" 검색 → 유사한 텍스트만 반환
- 관련 초안, 게시물, 댓글 간 연결 불가능

**Maestro Graph RAG**:
- "AI 트렌드" 검색 → 트렌드 노드 발견
- 자동 탐색: 관련 초안 → 발행 게시물 → 댓글 반응까지 연결
- **7단계 검색 파이프라인**: 벡터 유사도 + 그래프 탐색 결합

### 자동 동기화 시스템

**문제**: 수동으로 임베딩 생성/관리 → 불일치, 중복 작업

**Maestro 해결책**:
1. **도메인 테이블 변경 감지** (Celery watcher, 30초~5분 주기)
2. **규칙 기반 요약 생성** (LLM 호출 없음)
3. **벡터 임베딩 자동 생성** (중복 방지 해시 체크)
4. **관계 간선 자동 구성** (외래키 + 비즈니스 규칙)
5. **가시성 확보** — 사이드카 메트릭과 그래프 탐색 카드에서 즉시 확인 가능, “그래프 RAG와 사이드카가 정상 동작한다”는 사실을 운영 중에도 체감
6. **이벤트 브로커** — Canonicalizer가 동기화 직후 `GraphRagRefreshEvent`를 발행해 Copilot SSE가 즉시 최신 카드/ROI를 재계산

**비즈니스 효과**:
- 엔지니어가 임베딩 로직 작성할 필요 없음
- 데이터 변경 즉시 검색 가능 (30초 이내)
- 관계 정보 자동 유지 → 맥락 있는 답변 제공

이제 실제 운영 환경에서도 새 Draft나 Playbook이 저장되면 수십 초 안에 그래프에서 해당 판단과 연관된 트렌드, 페르소나, 캠페인을 모두 따라가 볼 수 있습니다. Graph RAG와 사이드카 조합이 “작동 중”이라는 표현은 단순 선언이 아니라, 매일 수집되는 로그가 그래프 탐색과 BFF 응답에 곧바로 반영되고 있음을 뜻합니다.

## 3. 아키텍처 이점

### 예측 가능한 확장성

**전통적 방식**:
- 새 기능 = 새 서비스 로직 + 새 API + 새 DB 쿼리
- 복잡도 선형 증가 → 유지보수 부담 급증

**Maestro 방식**:
- 새 기능 = 기존 플로우 조합 + 어댑터 등록 (50줄)
- 복잡도 상수 유지 → 장기 개발 속도 유지

**측정 가능한 지표**:
- 134개 플로우, 25개 모듈
- 6개 어댑터로 복합 기능 지원
- 8개 도메인이 자동 Graph RAG 동기화
- **신규 체이닝 기능 추가 시 필요 코드: ~50줄**

### 지식 그래프 확장 패턴

**신규 도메인 추가 시**:
1. Watcher 작성 (~30줄)
2. Payload 빌더 작성 (~50줄)
3. 완료 → 자동 임베딩/검색 지원

**기존 시스템과 비교**:
- 수동 임베딩 관리: 수백 줄 + 스케줄링 로직
- Maestro: 80줄로 완전 자동화

---

## 4. 실제 제약사항

### 투명한 한계 공개

**어댑터 범위**:
- 전체 134개 플로우 중 6개 체이닝 패턴만 지원
- 대부분의 플로우는 독립 실행
- 특정 도메인(댓글↔초안, 트렌드↔초안)에 집중

**Graph RAG 통합**:
- 8개 도메인만 자동 동기화
- 스케줄, 메트릭 등 다수 테이블 미통합
- 간선 생성 규칙은 수동 코딩 필요

**LLM 의존성**:
- 고품질 콘텐츠는 여전히 LLM 우선
- Fallback 로직 존재하나 품질 차이 있음

---

## 5. 비즈니스 가치 정당화

### 개발 속도 유지

**측정 가능한 효과**:
- 복합 기능 개발 시간: 기존 대비 60% 단축 (어댑터 재사용)
- 신규 도메인 검색 통합: 1일 → 2시간 (Graph RAG watcher)

### 운영 안정성

**LLM 비용 최적화**:
- 데이터 변환 로직은 LLM 사용 안 함
- Graph RAG 요약은 규칙 기반 (LLM 호출 제로)
- LLM은 최종 콘텐츠 생성에만 사용

**시스템 견고성**:
- LLM 실패 시 자동 Fallback
- 플로우 조합 실패 시 개별 플로우로 대체
- 데이터 동기화는 멱등성 보장 (중복 처리 방지)

### 장기 유지보수성

**핵심 설계 원칙**:
> "조합은 추가보다 저렴하다"

**실무 효과**:
- 신규 기능 요청 시 기존 컴포넌트 먼저 검토
- 80% 경우 새 코드 없이 조합으로 해결
- 20%만 신규 플로우 작성 필요

---

## 결론: 지속 가능한 AI 제품 아키텍처

**핵심 메시지**:

Maestro는 "AI를 빠르게 추가"하는 대신, **"AI를 예측 가능하게 조합"**하는 방식을 선택했습니다.

**결과**:
- ✅ 복합 기능 개발 속도 유지
- ✅ LLM 비용 최적화 (규칙 기반 + 선택적 LLM)
- ✅ 관계 기반 검색으로 맥락 있는 답변
- ✅ 투명한 한계 인지 및 Fallback 전략

**한계도 명확히**:
- ❌ "완전 자동화" 아님 (어댑터/watcher 수동 작성)
- ❌ "제로 코드" 아님 (조합 가능한 구조일 뿐)
- ❌ "전역 적용" 아님 (특정 도메인에 집중)

이 아키텍처는 **"마법"이 아닌 "엔지니어링 선택"**입니다.
