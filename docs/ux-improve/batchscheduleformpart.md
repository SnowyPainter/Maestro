# BatchScheduleFormPart UX 개선 제안

현재 `BatchScheduleFormPart` 컴포넌트는 기능적으로는 완전하지만, 사용자 경험(UX) 관점에서는 개선의 여지가 많습니다. UI가 시스템의 데이터 구조(모델)에 너무 가깝게 설계되어 있어, 사용자가 자신의 "의도"를 시스템이 요구하는 "형식"에 맞춰 입력해야 하는 인지적 부담이 큽니다.

이 문서는 문제점을 진단하고, "사람의 의도 중심" 설계로 전환하기 위한 구체적인 개선안을 제안합니다.

---

## 1. 핵심 문제 진단

| 문제점 | 현재 증상 | 사용자에게 미치는 영향 |
| :--- | :--- | :--- |
| **데이터 필드 중심 설계** | 사용자가 날짜 범위, 요일(weekmask), 시간 세그먼트 등 기술적인 필드를 직접 채워야 합니다. | 사용자는 "무엇을 하고 싶은지"보다 "어떻게 입력해야 하는지"에 집중하게 되어 피로도가 높고 이탈하기 쉽습니다. |
| **모호한 스케줄 유형 구분** | '나중에 실행'과 '지금 실행'이 탭으로 구분되어 있지만, '지금 실행' 선택 시 즉각적인 피드백이 없어 상태 변화를 인지하기 어렵습니다. | 사용자는 자신의 선택이 제대로 반영되었는지 확신할 수 없으며, 특히 '지금 실행'이 즉시 제출되는 것인지 혼란스러워합니다. |
| **복잡한 고급 설정 (세그먼트)** | 시간대별 실행 횟수를 설정하는 '세그먼트' UI가 여러 입력 필드의 나열이라 직관성이 떨어지고 수정이 번거롭습니다. | 대부분 "하루에 N번" 정도만 설정하려는 사용자에게 과도한 복잡성을 요구하며, 설정 오류의 가능성을 높입니다. |
| **피드백 부재** | '지금 실행'을 선택하거나 스케줄을 설정한 후, 시스템이 이를 어떻게 이해했는지 요약해주거나 확인시켜주는 장치가 없습니다. | 사용자는 자신의 설정이 원하는 대로 동작할지 불안감을 느끼며, 결과가 나온 후에야 설정이 잘못되었음을 깨달을 수 있습니다. |
| **입력 제약 부족** | 종료일이 시작일보다 빠르거나, 시간이 논리적으로 맞지 않는 등 잘못된 값을 입력해도 막아주지 않습니다. | 사소한 입력 실수가 시스템 오류로 이어져 서비스 전체의 신뢰도를 저하시킵니다. |

---

## 2. 개선 컨셉: "의도 중심의 스케줄링"

> "언제부터 언제까지, 얼마나 자주 실행할지"를 사람이 생각하는 방식 그대로 UI에 표현하고, 시스템이 이를 해석해 기술적인 스케줄 데이터로 변환합니다.

---

## 3. 구체적인 개선 방안

### (1) 상위 구조 변경: 직관적인 선택과 요약

**현재**: `Tabs("지금/나중")` → `Tabs("단순/고급")`의 중첩 구조

**개선안**: **명확한 모드 선택 + 설정 요약 카드**

1.  **최상단에서 실행 방식을 명확히 선택하게 합니다.**
    *   `[ ] 지금 한 번 즉시 실행`
    *   `[x] 예약 실행`
2.  **'예약 실행' 선택 시, 설정 내용을 실시간으로 요약해주는 카드를 보여줍니다.**

    ```
    [요약 카드]
    🗓️ **기간**: 2025년 10월 10일 ~ 2025년 10월 24일
    🔁 **반복**: 평일만
    ⏰ **시간**: 오전 9시 ~ 오후 5시 사이, 하루 3회
    ```

    사용자는 이 요약 카드를 보고 자신의 설정이 올바른지 즉시 파악할 수 있으며, 각 항목을 클릭해 해당 설정을 수정할 수 있습니다.

### (2) '단순 설정'을 더 단순하고 강력하게

**현재**: 시작/종료일 입력 + 빈도 슬라이더

**개선안**: **사전 정의(Preset) 기반의 리드미컬한 UX**

| 항목 | 개선안 | 기대 효과 |
| :--- | :--- | :--- |
| **기간** | 2개의 날짜 입력 필드 대신 **하나의 캘린더에서 범위(Range)를 선택**하도록 변경합니다. | 시작일과 종료일의 관계를 시각적으로 파악하기 쉽고, 입력 단계가 줄어듭니다. |
| **빈도** | 슬라이더 대신 **선택 칩(Chip)**을 사용합니다. (`하루 1회`, `하루 3회`, `매시간`, `직접 입력...`) | 가장 흔한 시나리오를 빠르게 선택하고, 필요 시에만 고급 설정으로 넘어갈 수 있습니다. |
| **시간대** | "언제쯤 실행할까요?"라는 질문과 함께 **시간대 칩**을 제공합니다. (`하루 종일`, `업무 시간`, `오전`, `오후`) | 대부분의 사용자는 정확한 시간보다 대략적인 시간대를 선호하므로, 복잡한 세그먼트 설정을 피할 수 있습니다. |

### (3) '고급 설정' UI 개선

**현재**: 입력 필드 목록의 반복

**개선안**: **테이블(Table) 형식 또는 시각적 편집기 도입**

세그먼트 목록을 테이블 형태로 바꾸어 가독성을 높입니다.

```tsx
<Table>
  <TableHeader>
    <TableRow>
      <TableHead>시작 시간</TableHead>
      <TableHead>종료 시간</TableHead>
      <TableHead>실행 횟수</TableHead>
      <TableHead />
    </TableRow>
  </TableHeader>
  <TableBody>
    {segments.map((segment, index) => (
      <TableRow key={index}>
        <TableCell><Input type="time" ... /></TableCell>
        <TableCell><Input type="time" ... /></TableCell>
        <TableCell><Input type="number" ... /></TableCell>
        <TableCell><Button variant="ghost" size="icon"><Trash2 /></Button></TableCell>
      </TableRow>
    ))}
  </TableBody>
</Table>
<Button variant="outline" size="sm"><PlusCircle /> 추가</Button>
```

이를 통해 각 세그먼트의 관계를 명확히 보고 쉽게 추가/삭제할 수 있습니다.

### (4) '지금 실행' 흐름 명확화

'지금 실행'은 스케줄 설정이 아닌, **즉시 실행 액션**입니다. 따라서 탭이 아닌 별도의 버튼이나 명확한 선택지로 분리해야 합니다.

선택 시, "즉시 실행되며 약 5분 안에 완료됩니다."와 같은 확인 메시지를 보여주고, 제출 후에는 "요청이 제출되었습니다"라는 토스트(Toast) 피드백을 주어 사용자를 안심시켜야 합니다.

## 4. 제안 리팩토링 구조

컴포넌트를 역할에 따라 분리하여 재사용성과 유지보수성을 높입니다.

```tsx
// /features/schedules/components/ScheduleBuilder.tsx (새로운 메인 컴포넌트)
<ScheduleBuilder value={...} onChange={...}>
  <ScheduleModeToggle /> // "지금 실행" / "예약 실행" 선택

  {mode === 'schedule' && (
    <>
      <ScheduleSummary /> // (3.1)에서 제안한 요약 카드
      <ScheduleSimpleSettings /> // (3.2)의 Preset 기반 설정
      <Collapsible>
        <CollapsibleTrigger>고급 설정</CollapsibleTrigger>
        <CollapsibleContent>
          <ScheduleAdvancedSettings /> // (3.3)의 테이블 기반 세그먼트 편집기
        </CollapsibleContent>
      </Collapsible>
    </>
  )}

  {mode === 'now' && <ImmediateRunConfirmation />} // (3.4)의 즉시 실행 확인
</ScheduleBuilder>
```

## 5. 기대 효과

| 목표 | 개선 후 모습 |
| :--- | :--- |
| **인지 부하 감소** | 사용자는 자연스러운 흐름에 따라 몇 번의 클릭만으로 스케줄을 설정할 수 있습니다. |
| **오류 가능성 감소** | Preset과 시각적 피드백(요약 카드, 캘린더)을 통해 잘못된 설정을 할 가능성이 줄어듭니다. |
| **사용자 만족도 증가** | 시스템이 사용자의 의도를 "이해"하고 있다는 인상을 주어, 더 신뢰감 있고 만족스러운 경험을 제공합니다. |
| **확장성 확보** | 역할별로 분리된 컴포넌트는 향후 챗봇 인터페이스 등 다른 영역에서 재사용하기 용이합니다. |

이 제안을 바탕으로 `BatchScheduleFormPart`를 리팩토링하여 사용자 중심의 컴포넌트로 개선할 수 있습니다.
