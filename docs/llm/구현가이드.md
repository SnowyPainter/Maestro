# LLM 통합 구현 가이드

## 배경
- 사용자는 “트렌드 데이터를 요약해서 초안 만들어줘”, “댓글 반응 기반으로 새 초안 뽑아줘” 같은 자연어 요청을 던진다.
- 기존 의도 분류/슬롯 추출만으로는 `bff.trends.list_trends → internal.llm.invoke → drafts.create` 같은 다단계 플랜을 안정적으로 만들기 어려웠다.
- LLM 호출을 별도의 Flow(`internal.llm.invoke`)로 노출하면, NLP 파이프라인이 이 Flow를 신뢰도 있게 고르지 못해 “LLM 호출 → 후속 Draft 생성” 연결이 끊겼다.

## 아키텍처 결정 (Flow vs Adapter)
| 대안 | 장점 | 단점/트레이드오프 |
| --- | --- | --- |
| **별도 Flow (`internal.llm.invoke`)** | - 엔드포인트 공개로 재사용성 높음<br>- Prompt key, vars 등을 API 레벨에서 제어 가능 | - NLP가 Flow 후보를 제대로 골라주지 못하면 체인이 끊김<br>- 사용자 요청마다 Flow를 여러 단계로 나누어 구성해야 함 |
| **Adapter 내부에서 LLM 호출 (採用)** | - Planner가 `bff.trends.list_trends → drafts.create` 처럼 단일 체인만 만들어도 내부에서 LLM 호출까지 처리<br>- 기존 Flow/Slot/Router에 대한 침습 최소화 | - 범용 LLM 호출 API를 잃음<br>- Flow 밖에서 임의 프롬프트를 호출하기 어려움 |

> **결론**: “LLM을 호출시키는 Adapter” 전략을 채택.  
>  - `trends_to_draft_adapter` / `comments_to_draft_adapter` 가 LLM을 직접 호출하고, 실패 시 기본 템플릿으로 폴백.  
>  - 사용자 요청은 `bff.trends.list_trends` 또는 `internal.insights.list_comments` → `drafts.create` 만 연결하면 된다.

## Adapter 기반 호출 흐름
```
트렌드 조회 결과 ─┐
                   ├─ trends_to_draft_adapter ──┐
Planner payload ──┘                             ├─ DraftSaveRequest → drafts.create
댓글 리스트 ───────┐                             │
                   ├─ comments_to_draft_adapter ┘
```
1. Planner가 최소한의 Flow 체인만 구성한다 (예: `bff.trends.list_trends → drafts.create`).
2. 어댑터는 LLM 호출이 필요하다고 판단되면 `LLMService.instance().ainvoke(...)`를 직접 호출한다.
3. LLM이 실패하거나 데이터가 부족하면, 간단한 규칙 기반 IR을 만들어 Draft를 생성한다.

## 구현 체크리스트
- [x] `LLMService` / `PromptVars` / `PromptKey`에 트렌드·댓글 초안용 스키마 반영.
- [x] `trends_to_draft_adapter`에서 `_use_llm` 옵션으로 LLM 호출 가능 여부 제어, 성공 시 DraftIR 주입, 실패 시 포맷 fallback.
- [x] `comments_to_draft_adapter`도 동일 패턴으로 구현.
- [x] Planner / DagExecutor가 payload builder가 `await`able을 반환할 수 있도록 수정.
- [x] `internal.llm.invoke` Flow, 관련 slot mention 제거.

## 테스트 전략
1. **단위**  
   - 어댑터의 LLM 성공/실패/비어있는 데이터 케이스 테스트 (Mock `LLMService`).  
   - Fallback DraftIR 구조 검증.
2. **통합**  
   - `bff.trends.list_trends → drafts.create` 체인 실행 후 Draft 생성 확인.  
   - 댓글 기반 요청(`internal.insights.list_comments → drafts.create`) 실행으로 폴백 포함한 응답 확인.
3. **회귀**  
   - Planner가 세그먼트 분할 없이도 다단계 체인을 만들 수 있는지 확인.  
   - Slot mentions/OpenAPI 등에서 `internal.llm.invoke` 제거로 인한 영향 검토.

## 향후 개선 아이디어
- 어댑터별 LLM 호출 히스토리 로깅/모니터링 추가 (Prompt, 응답 latency, 성공률).
- Prompt 버전 업/다국어 지원 시 어댑터에서 `prompt_key` 및 보조 파라미터를 전달하도록 옵션화.
- 나중에 범용 LLM 엔드포인트가 다시 필요해지면, Planner 레이어에서 “내장 어댑터 호출”과 “직접 Flow 호출”을 토글할 수 있는 플래그 도입 검토.***
