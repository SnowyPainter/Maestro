# Public Router 구현 교훈

## 문제 상황

ngrok은 기본적으로 1개의 터널링만 지원하는 제약이 있었고, 내부적으로 SeaweedFS (S3 호환 스토리지)를 사용하여 파일을 저장하고 있었다. 외부 서비스에서 이러한 미디어 파일들을 안전하게 접근할 수 있도록 프록시가 필요했다.

## FastAPI + httpx 스트리밍 핵심 교훈

### 1. async with 블록 밖에서 StreamingResponse 리턴 ❌
   - **문제**: async with 블록이 끝나면 스트림이 자동으로 닫히므로, StreamingResponse를 반환할 때 이미 스트림이 닫힌 상태가 된다.
   - **해결**: async generator 안에서 스트림을 열고 yield하여 데이터를 전송해야 한다.

### 2. StreamingResponse는 지연 전송
   - **특징**: generator가 yield할 때만 실제 데이터 전송이 이루어진다.
   - **중요성**: 메모리 효율적이며, 대용량 파일 전송에 적합하다.

### 3. Content-Length 헤더는 제거해야 안전
   - **문제**: 서버가 잘못된 길이 정보를 보내면 FastAPI가 오류를 발생시킬 수 있다.
   - **해결**: Content-Length 헤더를 명시적으로 제거하여 안전한 스트리밍을 보장한다.

### 4. HEAD 요청으로 헤더만 미리 가져오기
   - **장점**: 실제 데이터(body)를 전송하지 않고도 Content-Type, ETag 등의 메타데이터를 미리 확인할 수 있다.
   - **효율성**: 불필요한 데이터 전송을 방지하고 클라이언트에게 정확한 정보를 제공한다.

### 5. 생명주기 정리
   - **async with**: 스트림의 열기/닫기 범위를 명확히 정의
   - **yield**: 실질적인 데이터 전송 시점을 제어
   - **중요성**: 리소스 누수를 방지하고 안정적인 스트리밍을 구현

## 구현 시 주의사항

- `_FORWARDED_HEADERS` 세트에 필요한 헤더만 선택적으로 전달
- 타임아웃 설정으로 무한 대기 방지 (30초 스트리밍, 5초 헤드 요청)
- HTTP 상태 코드에 따른 적절한 예외 처리
- URL 인코딩으로 특수문자 처리
