# LLM 사용 시점과 아키텍처 설계

## 1. LLM이 꼭 필요한 지점

### 지침/콘텐츠 생성 (Generator 단계)

- **언어적 창의성 필요**: "어떻게 쓸지, 어떤 톤과 스타일로 만들지"
- **위험 요소**: A·B·C 기능처럼 결정론적 조합/실행은 오히려 LLM 쓰면 위험
- **적용 범위**: `Injector → LLMInterface → Draft/Guideline` 생성 구간에서만 LLM 활용

## 2. LLM이 필요 없는 지점

### 플래닝(Plan 생성)
- **처리 방식**: 어떤 기능을 실행할지, 순서, 선행조건 → 이미 DSL/블루프린트/룰 엔진으로 안정적으로 처리 가능
- **의도 분류**: 룰/파서/경량 모델로 충분

### 실행(Execution)
- **오케스트레이션**: Celery DAG / Temporal / BPMN 등 결정론적 오케스트레이터가 담당

### 검증/승인/롤백
- **기반**: 정책과 상태머신 기반. 언어적 창의성 전혀 필요 없음

## 3. 이렇게 분리하면 좋은 점

| 장점 | 설명 |
|------|------|
| **안정성** | 플랜은 항상 예측 가능. LLM이 엉뚱한 DAG 만들어서 사고낼 위험 없음 |
| **비용 절감** | 토큰 쓰는 부분은 지침/콘텐츠 생성으로 제한 → 훨씬 싸짐 |
| **재현성** | 같은 입력이면 항상 같은 실행 경로. 운영/감사 추적에 유리 |
| **안전** | 사이드이펙트(게시/삭제 등)는 LLM에게 절대 맡기지 않음 |

## 결론적으로

> **오케스트레이터는 LLM 사용하지 말 것.**

### 핵심 컴포넌트

- **Action Registry**: A/B/C 같은 원자 액션의 스키마·선행조건·보상 정의
- **Blueprint(고정 DAG)**: `SEQUENCE`/`ALL_OF`/`ANY_OF`/`APPROVAL_GATE` 컴비네이터로 작업 뼈대
- **Planner**: 슬롯 주입→정적 검증→부분실행(prune)→Celery chain/group/chord 합성
- **Saga(optional)**: 실패 시 보상(compensation) 실행
- **Idempotency**: `plan_hash + idempotency_key`로 중복 실행 방지

## 의도 분류 및 처리 방식

### 1. 행위 지시 (Action Intent)

키워드 기반 의도 매핑 가능:

| 사용자 입력 | 매핑된 액션 |
|-------------|-------------|
| "발행해줘" | `publish` |
| "예약해" | `schedule` |
| "삭제해" | `delete` |
| "수정해" | `edit` |

> **한국어 처리**: 띄어쓰기·어미 변화 때문에 조금 까다롭지만, 형태소 분석기(MeCab-ko, Kiwi, khaiii 등) + 사전 매칭으로 충분히 안정적임.

### 2. 콘텐츠 지침 (Guideline Intent)

- **LLM 필요**: 여기서는 결국 LLM 필요
- **라우팅 규칙**: "LLM을 호출해야 하는지 여부"는 규칙 분류로 잡을 수 있음
- **트리거 키워드**: "톤/스타일/해시태그/자연스러운 문장" 같은 키워드가 들어오면 → `Generator(LLMInterface)`로 라우팅
- **기본 처리**: 나머지는 다 결정론적 액션으로 처리

### 3. 메타/정보 조회 (Query Intent)

- **트리거 키워드**: "보여줘/어땠어/확인/리포트/조회" 같은 키워드 잡으면 → 조회 블루프린트 실행
- **처리 방식**: SQL 쿼리나 `Adapter.metrics` 호출로 매핑

### 4. 시간 인식 (Date/Time Parsing)

이 부분이 진짜 중요한데, 이미 범용 라이브러리로 해결 가능함:

#### Python 라이브러리

- **`dateparser`**
  - "내일 10시", "다음 주 화요일 오전" 같은 한국어 자연어 시간 파싱 가능
- **`parsedatetime`**
  - "tomorrow 10am", "next monday" 처리

#### 한국어 지원

- **`dateparser`**는 "2025년 9월 20일", "내일", "모레 오후 3시" 등 잘 됨
- **확장 가능**: 안 되면 사전 룰 추가 가능 (예: "주말" → 토/일)
